# 通用项目文档系统规范

## 1. 文档目标
- 文档系统是代码背后的稳定事实抽象。
- 文档系统服务两类读者：人类工程师和 AI coding agent。
- 文档系统让读者在有限上下文内快速理解项目职责、模块边界、数据结构、接口协议、状态流转、错误收敛、测试口径和核心决策。
- 文档系统不替代代码。实现细节由代码表达，稳定事实由文档表达。

## 2. 核心理念

### 2.1 稳定事实
- 文档只记录长期成立的事实。
- 稳定事实包括业务规则、模块职责、依赖边界、数据主事实、接口协议、状态语义、错误处理、测试验收和核心决策。
- 文档不记录函数内部实现步骤、局部变量、临时分支、私有算法细节和重复的代码字段清单。
- 当实现细节变化不改变稳定事实时，文档保持不变。
- 当协议、状态、数据结构、模块边界、错误语义或核心决策变化时，文档同步变化。

### 2.2 AI 上下文友好
- 文档使用高密度、短句、确定表达。
- 每篇文档先说明职责和边界，再说明事实、流程和验收。
- 文档用代码入口定位事实来源，避免复制代码定义。
- 文档保留系统不变量、异常路径和恢复规则，帮助 AI 在修改代码前建立正确上下文。

### 2.3 单跳可达
- `spec/00_INDEX.md` 是唯一导航入口。
- `spec/00_INDEX.md` 指向所有一级事实入口。
- 每篇文档只引用直接相关文档和代码事实入口。
- 任一事实最多通过一次跳转抵达：从索引到文档，或从文档到代码入口。
- 跨文档引用只服务直接依赖，不形成连续追链。

### 2.4 单一事实源
- 代码中的类型、schema、IDL、迁移文件和协议对象是字段与结构定义的事实源。
- 文档描述字段语义、状态含义、业务约束、写入路径、读取路径和变更影响。
- 文档不复制完整字段清单。
- 文档中的协议、状态和数据事实必须能回到明确代码入口。
- 文档与代码冲突时，先修正不一致来源，再提交变更。

## 3. 总体约束
- 不使用流程图。
- 不使用表格。
- 不使用 emoji。
- 不粘贴具体实现代码。
- 允许出现字段名、接口名、状态名、枚举名、文件名和目录名。
- 表达使用确定语气。
- 标题直接描述主题。
- 文档面向当前真实系统，不为假想需求预留复杂结构。

## 4. 目录结构
- 项目根目录建立 `spec/`。
- `spec/00_INDEX.md` 是文档索引和事实导航入口。
- `spec/SYSTEM_OVERVIEW.md` 记录系统职责、事实分层、运行时分层和依赖边界。
- `spec/SYSTEM_FLOWS.md` 记录主流程、恢复流程和异常收敛流程。
- `spec/EXTERNAL_BEHAVIOR.md` 记录外部使用者能观察到的行为、协议语义和限制边界。
- `spec/INTERNAL_BEHAVIOR.md` 记录内部协作约束、状态不变量、并发规则和调度约束。
- `spec/ERROR_HANDLING.md` 记录错误分类、降级、补偿、重试、恢复和用户侧表现。
- `spec/DECISION_LOG.md` 记录当前生效的核心技术和产品决策。
- `spec/TEST.md` 记录测试分层、断言模型、测试入口和验收口径。

以下为服务整体模块划分，可以参考但不需要完全照搬。目标是保证可读性和扩展性，模块高内聚、低耦合。
- `spec/SERVICE/` 记录应用服务、用例编排、后台任务和主链路模块。
- `spec/DOMAIN/` 记录领域模型、领域状态、纯规则和业务不变量。
- `spec/API/` 记录 HTTP、RPC、CLI、事件入口等外部接口。
- `spec/STORE/` 记录数据库表、集合、队列、缓存 key 和持久化主事实。
- `spec/INFRA/` 记录配置、运行时、数据库连接、消息中间件、观测、外部 provider 和生命周期。
- `spec/INTEGRATIONS/` 记录第三方系统、上游下游协议和适配边界。
- `spec/TOOLS/` 记录工具目录、工具执行、工具结果协议和工具安全边界。
- `spec/WORKERS/` 记录常驻 worker、定时任务、恢复任务和清理任务。

## 5. 索引规范

### 5.1 `00_INDEX.md` 职责
- 说明文档系统目标。
- 说明全局文档规则。
- 列出项目主要文件结构、模块划分以及划分原则。
- 列出每个子目录的事实入口。
- 列出关键代码事实入口。
- 保持当前实际存在文档的完整导航。

### 5.2 索引写法
- 每条索引包含文档路径和一句职责说明。
- 职责说明只描述该文档回答的问题。
- 索引不展开正文内容。
- 索引不记录历史废弃文档。
- 删除文档时同步删除索引项。
- 新增文档时同步新增索引项。

## 6. 文档类型

### 6.1 系统总览
- 记录系统对外承担的职责。
- 记录核心事实分层。
- 记录运行时分层。
- 记录模块依赖边界。
- 记录关键技术边界。
- 记录核心代码事实入口。

### 6.2 系统流程
- 按运行阶段描述主链路。
- 描述恢复链路和补偿链路。
- 描述异常收敛链路。
- 描述流程中的事实写入点和副作用边界。
- 描述幂等、顺序、并发和请求收口规则。

### 6.3 外部行为
- 记录外部调用方能观察到的行为。
- 记录请求、响应、事件、错误和限制的语义。
- 记录外部协议的稳定约束。
- 记录验收口径。

### 6.4 内部行为
- 记录内部模块协作方式。
- 记录状态不变量。
- 记录调度、锁、事务、缓存和恢复约束。
- 记录内部边界被破坏时的风险。

### 6.5 模块文档
- 记录模块职责。
- 记录代码入口。
- 记录对外接口。
- 记录核心流程。
- 记录状态、幂等和并发规则。
- 记录错误收敛。
- 记录观测与验收。
- 记录相关测试入口。

### 6.6 数据文档
- 记录数据对象承担的主事实。
- 记录字段口径和权威定义来源。
- 记录写入路径。
- 记录读取路径。
- 记录索引、唯一性、顺序和一致性约束。
- 记录注意点和相关测试。

### 6.7 接口文档
- 记录接口用途和调用边界。
- 记录请求协议和响应协议的代码事实入口。
- 记录认证、幂等、限流、错误和兼容约束。
- 记录外部可见行为和验收场景。

### 6.8 错误处理文档
- 记录错误来源。
- 记录错误分类。
- 记录系统收敛方式。
- 记录补偿和降级规则。
- 记录重试边界。
- 记录用户侧或调用侧最终表现。
- 记录错误事实的排障入口。

### 6.9 决策记录
- 记录当前生效决策。
- 每条决策包含决策内容、原因、代码影响、测试影响、排障影响和状态。
- 决策记录只保留对当前系统仍有约束力的内容。
- 废弃决策保留废弃状态和替代决策入口。

### 6.10 测试文档
- 记录测试目标。
- 记录测试分层。
- 记录资源隔离规则。
- 记录公共测试支撑层。
- 记录断言模型。
- 记录测试定位入口。
- 记录禁止执行的测试方式及原因。

## 7. 标准模板

### 7.1 模块文档模板
- 标题使用模块名。
- `模块职责` 说明模块负责什么、不负责什么。
- `代码入口` 列出稳定入口文件、类型或对象。
- `对外接口` 描述其他模块调用本模块的入口。
- `核心流程` 按事实变化顺序描述主路径。
- `状态与幂等` 描述状态来源、状态转移、幂等键和顺序规则。
- `错误收敛` 描述失败后的事实写入、降级、重试和资源回收。
- `观测与验收` 描述日志、指标、检查点和验收标准。
- `相关测试` 列出测试目录、测试族或测试定位脚本。

### 7.2 数据文档模板
- 标题使用数据对象名。
- `职责` 说明该数据对象保存的主事实。
- `字段口径` 指向代码事实入口，说明关键字段语义和权威值域。
- `当前语义` 说明当前业务解释和状态含义。
- `写入路径` 说明哪些模块写入该数据对象。
- `读取路径` 说明哪些模块读取该数据对象。
- `约束` 说明唯一性、索引、顺序、一致性和事务要求。
- `注意点` 说明容易误解的边界。
- `相关代码与测试` 列出事实入口和测试入口。

### 7.3 接口文档模板
- 标题使用接口名。
- `职责` 说明接口解决的问题。
- `协议入口` 指向请求、响应、事件和错误类型定义。
- `请求语义` 描述调用方必须提供的稳定信息。
- `响应语义` 描述调用方会观察到的稳定结果。
- `错误语义` 描述失败状态、错误码、重试和降级。
- `幂等与顺序` 描述重复调用和并发调用的处理规则。
- `验收场景` 描述必须覆盖的正常、异常和边界场景。

### 7.4 决策记录模板
- `决策` 说明已经采用的方案。
- `原因` 说明该方案解决的核心问题。
- `代码影响` 说明受约束的模块、边界和入口。
- `测试影响` 说明必须覆盖的测试方向。
- `排障影响` 说明出现问题时的定位入口。
- `状态` 使用 `生效`、`废弃` 或 `被替代`。

## 8. 写作规则
- 先写结论，再写约束，再写细节。
- 每个段落只表达一个事实。
- 同一概念只使用一个名称。
- 状态、枚举、字段和接口名使用代码中的真实名称。
- 模块职责使用正向描述，边界使用明确排除。
- 错误处理必须说明事实是否写入、是否重试、是否补偿、外部是否可见。
- 流程描述按事实变化顺序排列。
- 文档不使用营销表达、情绪表达和模糊判断。
- 文档不写“后续完善”“按需扩展”“灵活处理”等空泛内容。

## 9. 代码事实入口规则
- 每篇文档必须列出与正文事实直接相关的代码入口。
- 代码入口指向稳定类型、schema、协议对象、迁移文件、配置对象、服务入口或测试入口。
- 文档不引用临时 helper 和纯私有细节作为事实源。
- 字段定义以类型、schema、IDL 或迁移文件为准。
- 状态值域以枚举、联合类型或协议定义为准。
- 测试入口以测试目录、测试族或测试定位脚本为准。

## 10. 维护流程

### 10.1 代码变更前
- 先阅读 `spec/00_INDEX.md`。
- 按变更范围读取相关事实文档。
- 定位协议、状态、数据、流程、错误和测试是否受影响。
- 修改前明确需要同步更新的文档。

### 10.2 代码变更中
- 新增外部接口时更新 `API` 或 `INTEGRATIONS` 文档。
- 新增数据对象时更新 `STORE` 或 `DOMAIN` 文档。
- 新增状态或枚举时更新对应模块文档、数据文档和错误处理文档。
- 新增服务模块时更新 `SERVICE` 文档和 `00_INDEX.md`。
- 新增 worker 时更新 `WORKERS` 文档和恢复或错误处理文档。
- 改变错误语义时更新 `ERROR_HANDLING.md` 和相关模块文档。
- 改变测试分层、测试入口或断言口径时更新 `TEST.md`。
- 做出核心架构或产品决策时更新 `DECISION_LOG.md`。

### 10.3 代码变更后
- 检查文档与代码事实入口一致。
- 检查索引包含新增、删除和重命名文档。
- 检查相关测试入口仍然有效。
- 检查文档没有复制实现代码。
- 检查文档没有引入连续引用链。

## 11. 质量门禁
- `spec/00_INDEX.md` 能定位全部事实入口。
- 每个模块都有明确职责、边界、代码入口和测试入口。
- 每个外部接口都有协议事实入口、错误语义和验收场景。
- 每个数据主事实都有权威定义来源、读写路径和一致性约束。
- 每个核心状态都有来源、值域、转移规则和非法状态处理方式。
- 每个错误类别都有收敛、降级、重试和外部表现。
- 每个核心决策都有原因、影响和状态。
- 文档没有流程图、表格、emoji 和实现代码片段。
- 文档没有与代码事实冲突的重复字段定义。

## 12. 新项目落地步骤
- 创建 `spec/` 目录。
- 创建 `spec/00_INDEX.md`。
- 创建 `spec/SYSTEM_OVERVIEW.md`。
- 创建 `spec/SYSTEM_FLOWS.md`。
- 创建 `spec/ERROR_HANDLING.md`。
- 创建 `spec/DECISION_LOG.md`。
- 创建 `spec/TEST.md`。
- 按真实边界创建 `SERVICE`、`DOMAIN`、`API`、`STORE`、`INFRA`、`INTEGRATIONS`、`TOOLS` 和 `WORKERS` 文档。
- 在索引中登记所有实际存在的文档。
- 在每篇文档中登记直接相关代码事实入口。
- 在每次代码变更中同步维护受影响文档。

## 13. 适用边界
- 本规范适用于持续迭代的编程项目。
- 本规范适用于单体、模块化单体和分布式系统。
- 本规范适用于后端服务、前端应用、CLI 工具、数据管道、AI 应用和基础设施项目。
- 本规范不替代产品 PRD、部署手册、运行手册和用户手册。
- 当其他文档承担运行操作或用户培训时，`spec/` 仍只记录代码背后的稳定事实。
